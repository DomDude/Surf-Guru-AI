# Surf Guru AI — Task Plan

> **How this file works.** Work is organized in **5 phases**, each split into **parallel tracks** (A/B/C/D/E). A track is a column of tasks a single Claude Code session can own end-to-end without colliding with other tracks. Every task lists (a) the files it owns, (b) the files it must not touch, (c) the acceptance criteria, and (d) its dependencies. If you're in a fresh session: read `CLAUDE.md` first, pick one track, finish it, update the checkbox, append to `progress.md`, and note anything subtle in `findings.md`.
>
> **Parallel-session safety:** tasks within the same phase marked `[P]` can run simultaneously in different sessions. Tasks marked `[S]` are serial — wait for their stated dependency before starting. Phases are roughly sequential, but later-phase tasks marked `[Pre]` can start early once their prerequisites land.
>
> **Legend:** `[ ]` todo · `[~]` in progress · `[x]` done · `[!]` blocked (add reason)

## Phase 3 — Intelligence Layer (The Wedge)
**Goal:** Make the AI *actually* intelligent. Agent pipeline, local knowledge ingestion, stoke score, topography reasoning, skill-aware gating.
**Exit criteria:** `route.ts` is a plan/gather/reason/render pipeline. At least one external knowledge source is live. Stoke score is computed deterministically with LLM modifiers. Intermediate surfers are never recommended suicidal conditions.

### Track 3A — Agent pipeline refactor [S, blocks 3B–3D on shared types]
Owns: `webapp/src/app/api/chat/route.ts`, new `webapp/src/lib/agent/*`.
Do not touch: tools (they stay pure).

- [ ] **3A.1 Define the pipeline stages** [S]
  - `plan(input)` → `{ spots: SpotCoordinates[]; sources: string[] }` (trivial for single-spot; rich for region ranker).
  - `gather(plan)` → `{ forecasts, tide, local_knowledge, agreement }`.
  - `reason(gathered, user)` → `{ stoke_score, reasoning, recommendations }`.
  - `render(reasoned)` → final payload matching the `gemini.md` schema.
  - Each stage is a separate file in `src/lib/agent/`.
  - Acceptance: existing single-spot flow passes through all four stages with identical UX.

- [ ] **3A.2 Pass structured context to Gemini instead of a pasted JSON blob** [S, after 3A.1]
  - System prompt stays short and persona-focused.
  - User message contains the structured gathered data as clearly-labeled sections.
  - Acceptance: Gemini output quality equal or better on 3 test spots.

### Track 3B — Stoke score [P, after 3A.1]
Owns: new `webapp/src/lib/stoke.ts`, `gemini.md` (add schema).
Do not touch: tools.

- [ ] **3B.1 Deterministic base formula** [S]
  - Input: ForecastPoint, spot metadata (from gazetteer), skill level.
  - Factors: swell size fit to skill, period quality, wind offshore-ness (relative to spot's `best_wind_dir_deg`), swell direction fit to `best_swell_dir_deg`, model agreement.
  - Output: 0–100 integer, plus per-factor breakdown for explainability.
  - Calibrate against: known-epic (Supertubos big groundswell, light E wind) → 85+; known-flat (summer Sagres, S wind) → <30.
  - Acceptance: hand-tuned against 5 known conditions, all within ±10 of expected.

- [ ] **3B.2 LLM modifier pass** [S, after 3B.1]
  - Gemini sees the base score and factor breakdown and may adjust ±10 with a written reason.
  - Never allowed to flip category (good ↔ poor).
  - Acceptance: adjustments are bounded and always come with a reason.

- [ ] **3B.3 Expose score in payload + UI** [S, after 3B.2]
  - Add `stoke_score: { value, factors, modifier, modifier_reason }` to the payload schema in `gemini.md`.
  - UI shows the score prominently, with a collapsible "Why?" showing factors.
  - Acceptance: visible, explainable, non-hallucinated score on the report.
